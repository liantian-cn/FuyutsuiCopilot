# Fuyutsui 文档审核

多代理（Multi-Agent）协作审核流程，系统性审查 Fuyutsui 项目的中文说明文档。

## 前置条件

- 用户必须指定要审核的文档目录名称（如 `玩家光环`、`技能冷却`、`截图原理`）
- 该目录下必须存在 `readme.md`
- **绝不**修改 `Fuyutsui/` 目录下的任何源码文件

## 执行方式

从用户消息中提取文档目录名称（`docName`），然后调用 Workflow 工具，使用项目根目录下预存的脚本文件：

```
Workflow({ scriptPath: '.claude/workflows/fuyutsui-doc-review.js', args: { docName: "提取到的目录名" } })
```

该脚本位于 `.claude/workflows/fuyutsui-doc-review.js`，是已通过解析验证的纯 JavaScript 脚本（使用函数拼接代替模板字面量中的反引号转义，避免解析错误）。

如果用户没有指定文档名称，先询问用户要审核哪个文档。

## 文档写作原则

**描述原理，不标注行号。** 文档面向 mod 开发者，目标是解释代码「做了什么、为什么这样做」，而不是「代码在第几行」。行号每次提交都会变化，对读者毫无意义。

- **移除旧行号**：审核修改时主动发现并移除文档中已有的行号引用（如「第 42 行」「L128」），用代码结构定位替换。
- **使用调用链定位**：引用源码时使用 `文件名 > 类名 > 函数名 > 子函数名` 格式（如 `core/bar.lua > BarManager > UpdateCooldown > CheckCharge`），让读者在代码树中定位而非在屏幕坐标上定位。
- **最短路径原则**：省略不存在的层级，目标是从文件到关键逻辑的最短无歧义路径。
- **描述行为**：某个函数/事件/模块负责什么、消费什么数据、产出什么结果。若涉及多文件只需描述调用链。

此原则贯穿全部审核阶段（Alpha→Beta→Gamma 审查、Epsilon→Zeta→Eta 复查、Theta 综合、Iota 修改）。

## 结果

Workflow 完成后，文档 `{文档名称}/readme.md` 已被修改。总结以下信息告知用户：
- 发现的问题数量和严重程度分布
- 实际修改了多少处
- 建议用户使用 `git diff` 查看具体变更
